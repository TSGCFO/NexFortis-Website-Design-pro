## CRITICAL ISSUE — CORS Blocks All API Calls on Temporary Render URLs

### What happened

The agent introduced `VITE_API_BASE_URL=https://nexfortis-api.onrender.com` and baked it into both static site bundles at build time. This means:

- **Marketing site:** `apiFetch("/contact")` now calls `https://nexfortis-api.onrender.com/api/contact` directly (cross-origin)
- **QB portal:** `getApiBase()` returns `https://nexfortis-api.onrender.com`, so every single fetch call goes cross-origin

The API server's CORS configuration (in `artifacts/api-server/src/app.ts`, lines 52-64) only allows these origins:
- `https://nexfortis.com`
- `https://www.nexfortis.com`
- `https://qb.nexfortis.com`
- `https://qbportal.nexfortis.com`

It does **NOT** allow:
- `https://nexfortis-marketing.onrender.com`
- `https://nexfortis-qb-portal.onrender.com`

**Result:** Every API call from both frontends will fail with a CORS error on the current temporary Render URLs. Contact form, QB portal login, checkout, orders, tickets, waitlist — everything is broken.

### Why the rewrite approach was correct

Before these commits, the render.yaml had rewrite rules like:
```yaml
routes:
  - type: rewrite
    source: /api/*
    destination: https://nexfortis-api.onrender.com/api/:splat
```

The `:splat` syntax was wrong (that's Netlify, not Render), and commit d37323c correctly fixed it to `*`. **But then the agent also added VITE_API_BASE_URL**, which makes the code bypass the rewrites entirely.

The rewrite approach is the RIGHT architecture because:
- Rewrites happen at Render's CDN level — the browser sees the request going to the same origin
- No CORS issues at all
- No need to add temporary origins to the CORS allow list
- Works identically on temporary and custom domain URLs

### What needs to happen

**Option A (recommended):** Remove `VITE_API_BASE_URL` from render.yaml and revert the frontend code to use relative `/api/` paths. Keep the fixed rewrite rules (`*` instead of `:splat`). This is the simplest and most correct approach.

**Option B:** Add the temporary Render origins to the CORS allow list. This works but is messy — you'd need to remove them again once custom domains are set up, and it means maintaining two sets of origins.

## VERDICT: ❌ DOES NOT PASS

### The one critical issue

The `VITE_API_BASE_URL` approach introduces a CORS blocker that breaks every API call from both frontends on the current Render URLs. The rewrite fix (`:splat` → `*`) was the correct and complete solution by itself.

### What's good and should be kept

1. The `:splat` → `*` rewrite fix in render.yaml — this was the core fix we needed
2. The `getApiBase()` / `apiUrl()` centralization pattern in the QB portal — excellent refactor that eliminates 10+ instances of duplicated `BASE_URL` parsing
3. The flat `.html` mirror writes in both prerender scripts — correct solution for Render's clean URL handling
4. The redirect rule ordering in render.yaml

### Recommended fix path

1. **Remove `VITE_API_BASE_URL` from render.yaml** for both static sites (don't set it at all)
2. **Keep `getApiBase()` and `apiUrl()` code as-is** — when `VITE_API_BASE_URL` is not set, these correctly fall back to relative paths that go through the rewrites
3. **Redeploy both static sites** so the bundles are built without the env var, and all fetch calls use relative `/api/` paths through the rewrites
4. **Later (Phase 2):** Once custom domains are set up (`nexfortis.com` and `qb.nexfortis.com`), the CORS allow list already includes them, so you can optionally set `VITE_API_BASE_URL=https://api.nexfortis.com` at that point. But even then, the rewrite approach is cleaner.

This way we keep all the good code quality improvements while using the correct architecture (server-side rewrites) that avoids CORS entirely.
