## BLOCKING ISSUES (must fix before merge)

### B1. Path Traversal in Both Prerender HTTP Servers

**Both reviewers agree. Confirmed by manual review.**

- **Files:** `artifacts/nexfortis/prerender.mjs:128` and `artifacts/qb-portal/prerender.mjs:118`
- **Problem:** The local HTTP server used during the build step takes the incoming URL and passes it directly to `path.join(distDir, url)`. Since `path.join()` resolves `..` segments, a request like `GET /../../../etc/passwd` would escape the `dist/public` folder and serve any file on the machine. While the server only binds to `127.0.0.1` and runs briefly, any other process on the same machine (including Puppeteer's Chromium) could exploit this. In shared CI runners (like Replit), this is a real risk.
- **Fix needed:** After computing `filePath`, verify it stays inside `distDir`:

```js
const resolved = path.resolve(distDir, url.slice(1));
if (!resolved.startsWith(distDir + path.sep) && resolved !== distDir) {
  res.writeHead(403);
  res.end("Forbidden");
  return;
}
```

### B2. Shell `noindex` Tag Survives if Prerender Silently Fails

**Found by GPT-5.4 reviewer. Confirmed by manual review.**

- **Files:** `artifacts/nexfortis/index.html:14`, `artifacts/qb-portal/index.html:11`, and both `prerender.mjs` verification functions
- **Problem:** Both `index.html` shells contain `<meta name="robots" content="noindex, nofollow" />`. This is intentional — the shell shouldn't be indexed. The prerender script replaces each page's HTML with a version where React's `<meta name="robots" content="index,follow,...">` tag overrides the shell one via the `dedupeSeoTags()` function (last-wins for meta tags).

  **The danger:** If a prerender error gets caught silently (the `catch` block on line 207/197 increments `fail` but doesn't exit until after the loop), or if a new route gets added to `App.tsx` without being picked up by the route-discovery regex, the SPA shell (with `noindex`) would be served to search engines for that page. The `verifyPrerendered()` function only checks that an `index.html` file exists on disk — it does **not** read the file content to confirm the `noindex` was actually replaced.

  This means a page could be fully invisible to Google with zero indication that anything is wrong.

- **Fix needed:** Add content-level verification in `verifyPrerendered()`:

```js
const content = fs.readFileSync(file, "utf-8");
if (content.includes("noindex") && !NOINDEX_ROUTES.includes(route)) {
  missing.push(`${route} (still contains noindex)`);
}
```

## IMPORTANT ISSUES (should fix before merge or immediately after)

### I1. SearchAction URL Templates Reference Non-Existent Search Features

**Both reviewers agree. Confirmed by manual review.**

- **Files:** `artifacts/nexfortis/src/components/seo.tsx:156` and `artifacts/qb-portal/src/lib/seo-schemas.ts:41`
- **Problem:** The `WebSiteSchema` in both apps declares a `SearchAction` with `?q={search_term_string}`:
  - nexfortis points to `/blog?q={search_term_string}` — the blog page doesn't read or use a `?q=` parameter
  - qb-portal points to `/catalog?q={search_term_string}` — the catalog page uses local React state (`useState`), not URL parameters
- **Impact:** Google validates these targets. When users search via the sitelinks search box and land on the page, the `?q=` parameter is silently ignored. Google may stop showing the sitelinks search box after repeated failures.
- **Recommendation:** Remove the `potentialAction` / `SearchAction` from both schemas until URL-parameter-driven search is implemented. Or wire both pages to read `?q=` from the URL.

### I2. Title Dedup Logic Is Keep-First (Should Be Keep-Last)

**Both reviewers agree on the risk. Confirmed by manual review.**

- **File:** `lib/seo-dedupe.mjs:70-71`
- **Problem:** The dedup logic keeps the **first** `<title>` tag and the **last** of everything else. This means if the shell `<title>NexFortis IT Solutions</title>` appears before React Helmet's page-specific title in the captured HTML, the generic shell title would survive instead of the page-specific one.
- **Mitigation:** In practice, Puppeteer's `page.content()` captures the final DOM state after `networkidle0` + 250ms, meaning React Helmet should have already replaced the shell title in-place. So there should be only ONE `<title>` in the captured HTML in normal operation. The risk materializes only under partial hydration or slow React init.
- **Recommendation:** Change to keep-last for safety (matching the meta tag behavior):

```js
// Instead of: if (!keepIdxByKey.has(mt.key)) keepIdxByKey.set(mt.key, i);
// Use: keepIdxByKey.set(mt.key, i);  // keep last for title too
```

### I3. Prerender Server Has No Error Handler for Port Conflicts

**Both reviewers agree. Confirmed by manual review.**

- **Files:** `artifacts/nexfortis/prerender.mjs:120-142` and `artifacts/qb-portal/prerender.mjs:110-132`
- **Problem:** The `startServer()` function wraps `server.listen()` in a Promise that only `resolve`s on success. If port 4173/4174 is already in use, `server.listen()` emits an `error` event, but there's no handler. The Promise hangs forever, causing the build to timeout instead of failing with a clear error.
- **Fix:** Add `server.on('error', (err) => reject(err))` before `server.listen()`.

### I4. `sharp` and `puppeteer` Should Be devDependencies

**Both reviewers agree. Confirmed by manual review.**

- **Files:** `package.json:26` (sharp), `artifacts/nexfortis/package.json:63` (puppeteer), `artifacts/qb-portal/package.json:65` (puppeteer)
- **Problem:** `sharp` (~30MB) and `puppeteer` (~400MB with Chromium) are listed as production `dependencies` but are only used at build time (badge conversion and prerendering). They'll be installed in production deployments unnecessarily, bloating the deployment.
- **Fix:** Move `sharp` to `devDependencies` in root `package.json`. Move `puppeteer` to `devDependencies` in both app `package.json` files.

### I5. Hreflang Dedup Collides Non-Hreflang Alternates

**Found by GPT-5.4 reviewer. Confirmed by manual review.**

- **File:** `lib/seo-dedupe.mjs:48-51`
- **Problem:** `<link rel="alternate">` tags without `hreflang` get keyed as `alternate:` (empty string). If there were multiple non-hreflang alternate links (e.g., RSS feed, JSON feed), only the last would survive. Currently no such tags exist, but this is a correctness gap.
- **Fix:** Only dedup `rel="alternate"` when `hreflang` is present; let non-hreflang alternates through undeduplicated.

## NON-BLOCKING ISSUES (fix in a follow-up)

### N1. SAFE_SLUG Regex Rejects Single-Character Slugs

**Both reviewers agree.**

- **File:** `artifacts/nexfortis/prerender.mjs:81`
- **Regex:** `/^[a-z0-9][a-z0-9-]*[a-z0-9]$/` requires minimum 2 characters
- **Impact:** Low — all current blog slugs are multi-word. But a future single-character slug would cause a `FATAL` exit.
- **Fix:** Change to `/^[a-z0-9]([a-z0-9-]*[a-z0-9])?$/`

### N2. Route Discovery Regex Is Fragile

**Found by GPT-5.4 reviewer. Confirmed.**

- **Files:** Both `prerender.mjs:45`/`prerender.mjs:53`
- **Regex:** `/<Route\s+path="([^"]+)"/g` requires `path` to be the first attribute after `<Route`
- **Impact:** If Prettier or a developer reorders attributes to `<Route component={X} path="..."`, the regex breaks and that route won't be prerendered.
- **Fix:** Change to `/<Route\s[^>]*path="([^"]+)"/g`

### N3. Browser Crash During Prerender Leaves Server Running

**Found by GPT-5.4 reviewer. Confirmed.**

- **Files:** Both `prerender.mjs` finally blocks
- **Problem:** If `browser.close()` throws (e.g., Chromium crash), `server.close()` never executes.
- **Fix:** Wrap `browser.close()` in try-catch:
```js
} finally {
  try { await browser.close(); } catch {}
  server.close();
}
```

## POSITIVE FINDINGS (things done well)

1. **Prerender pipeline is well-designed:** Route auto-discovery from App.tsx, blog API with checked-in fallback, sequential processing with individual page error handling, verification step after completion.

2. **SEO tag deduplication is solid:** The `dedupeSeoTags()` function correctly identifies and deduplicates `<title>`, `<meta>`, `<link rel="canonical">`, and `<link rel="alternate" hreflang="...">` tags using a keep-last strategy (except for title — see I2).

3. **JSON-LD schema structure follows Google's recommended patterns:** Organization, LocalBusiness, WebSite, Service, FAQ, Article, Breadcrumb, HowTo schemas are all properly structured with `@id` cross-references.

4. **`LocalBusiness @id` collision fixed:** The `LocalBusiness` schema now correctly uses `/#localbusiness` instead of `/#organization`, preventing JSON-LD entity collision.

5. **Lazy loading strategy in QB portal is clean:** SEO-facing pages (Home, Catalog, FAQ, etc.) are eagerly imported; auth/admin pages are lazy-loaded. The dual Suspense boundary architecture correctly isolates admin failures from customer pages.

6. **All page files have default exports:** Every file imported via `React.lazy()` has a proper `export default function`, so lazy loading will work correctly.

7. **URL consistency is correct:** All cross-site references use the correct domains (`nexfortis.com` vs `qb.nexfortis.com`).

8. **SPA fallback (`200.html`) is correctly handled:** The shell HTML (with `noindex`) is written to `200.html` after prerendering, ensuring catch-all routes serve the shell while prerendered routes serve indexed content.

9. **`isBuild` guard in vite configs is clean:** Environment variables (`PORT`, `BASE_PATH`) are only required during dev, not during build. This prevents build failures in CI where those vars aren't set.

10. **Geo meta tags present in both apps:** Both `nexfortis/seo.tsx` (lines 53-54) and `qb-portal/seo.tsx` (lines 82-83) include `geo.region` and `geo.placename` tags. (Note: Claude Code reviewer incorrectly flagged these as missing from nexfortis — they are present.)

11. **Prerender excluded routes match App.tsx routes:** All dynamic routes (`:slug`, `:id`), auth routes, admin routes, and the redirect route are properly excluded from prerendering.

12. **No race conditions in prerender loop:** Routes are processed sequentially with one page at a time. File writes are sequential. The `browser` instance is shared but pages are isolated.

13. **Token stripping in landing pages is defensive:** `resolveTokens()` and `stripTokens()` correctly remove unresolved `{token}` patterns and collapse whitespace, preventing broken-looking text.

14. **`convert-badges-to-webp.js` script is correct:** Properly converts PNG badge images to WebP across both apps with quality 80 and logging.
