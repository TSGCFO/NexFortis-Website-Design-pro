## Step 1: Database Schema тАФ Promo Code Tables

Extend the Drizzle schema in `lib/db/src/schema/qb-portal.ts` with four new tables and three new columns on the existing `qb_orders` table.

### 1a. `qb_promo_codes` Table

This is the master promo codes table. Columns:

- `id`: serial primary key
- `code`: text, not null, unique тАФ the customer-facing code string (stored uppercase, lookups are case-insensitive)
- `isActive`: boolean, not null, default true
- `type`: text, not null тАФ constrained to: `percentage`, `fixed_amount`, `free_service`, `subscription`. Add a CHECK constraint.
- `percentOff`: integer, nullable тАФ 1 to 100, used for percentage and free_service types
- `amountOffCents`: integer, nullable тАФ CAD cents, used for fixed_amount type
- `subscriptionDurationMonths`: integer, nullable тАФ used for subscription type only
- `maxUses`: integer, nullable тАФ null means unlimited
- `maxUsesPerCustomer`: integer, nullable тАФ null means unlimited per customer
- `redemptionCount`: integer, not null, default 0
- `expiresAt`: timestamp with time zone, nullable тАФ null means never expires
- `productIds`: text array, nullable тАФ specific product IDs this code applies to; null/empty means all products
- `categoryIds`: text array, nullable тАФ category slugs: `conversion`, `data_services`, `migration`, `subscription`, `volume_pack`; null/empty means all categories
- `minOrderAmountCents`: integer, nullable тАФ null means no minimum
- `firstTimeCustomerOnly`: boolean, not null, default false
- `stackableWithLaunchPromo`: boolean, not null, default true
- `restrictedToEmail`: text, nullable тАФ stored lowercase; null means no restriction
- `appliesToBasePrice`: boolean, not null, default false тАФ if true, stacked discount applies to base price, not post-launch-promo price
- `ownerUserId`: uuid, nullable, references `qb_users.id` тАФ set for referral codes
- `stripeCouponId`: text, nullable
- `stripePromotionCodeId`: text, nullable
- `description`: text, nullable тАФ internal operator notes
- `createdAt`: timestamp, not null, default now
- `updatedAt`: timestamp, not null, default now

> **тЪая╕П DO NOT:**
> - Use uuid for the `id` column тАФ keep it serial integer like all other `qb_*` tables
> - Forget the CHECK constraint on `type` тАФ it must restrict to the 4 valid values
> - Make `code` case-sensitive in the database тАФ add a unique index on `LOWER(code)` using a Drizzle raw SQL index

### 1b. `qb_promo_code_redemptions` Table

Records each time a code is successfully redeemed. Columns:

- `id`: serial primary key
- `promoCodeId`: integer, not null, references `qb_promo_codes.id`
- `userId`: uuid, nullable, references `qb_users.id` тАФ null for guest orders
- `guestEmail`: text, nullable тАФ populated for guest orders
- `orderId`: integer, not null, references `qb_orders.id`
- `discountAmountCents`: integer, not null
- `orderTotalBeforeCents`: integer, not null
- `orderTotalAfterCents`: integer, not null
- `ownerUserId`: uuid, nullable, references `qb_users.id` тАФ copied from the promo code's owner at redemption time
- `redeemedAt`: timestamp with time zone, not null, default now

### 1c. `qb_referral_credits` Table

Tracks credits earned by Premium subscribers through referral code redemptions. Columns:

- `id`: serial primary key
- `beneficiaryUserId`: uuid, not null, references `qb_users.id` тАФ the Premium subscriber earning the credit
- `amountCents`: integer, not null
- `source`: text, not null, default `promo_code_redemption`
- `promoCodeRedemptionId`: integer, nullable, references `qb_promo_code_redemptions.id`
- `orderId`: integer, nullable, references `qb_orders.id`
- `status`: text, not null, default `pending` тАФ constrained to: `pending`, `applied`, `paid_out`. Add a CHECK constraint.
- `createdAt`: timestamp, not null, default now

### 1d. `qb_promo_code_admin_events` Table

Audit log for all admin actions on promo codes. Columns:

- `id`: serial primary key
- `adminUserId`: uuid, not null, references `qb_users.id`
- `action`: text, not null тАФ `created`, `updated`, `deactivated`, `reactivated`
- `promoCodeId`: integer, not null, references `qb_promo_codes.id`
- `beforeState`: jsonb, nullable
- `afterState`: jsonb, nullable
- `createdAt`: timestamp, not null, default now

### 1e. Extend `qb_orders` Table

Add three new columns to the existing `qb_orders` table:

- `promoCodeId`: integer, nullable, references `qb_promo_codes.id`
- `discountAmountAppliedCents`: integer, nullable, default 0
- `subtotalBeforeDiscountCents`: integer, nullable
- `paymentStatus`: text, nullable тАФ add to support `free_promo` alongside existing statuses

### 1f. Create Insert Schemas

Add `createInsertSchema` exports for all four new tables, following the existing pattern in the schema file.

### 1g. Sync the Database

## Step 2: Promo Code Validation API

Create a new route file `artifacts/api-server/src/routes/qb-promo.ts` for promo code endpoints. Register it in the route index.

### 2a. `POST /api/qb/promo/validate`

This endpoint validates a promo code against an order without consuming it. It does NOT increment the usage counter.

**Request body:**
- `code` (string, required) тАФ the promo code entered by the customer
- `orderItems` (array, required) тАФ each item has `productId` (string), `quantity` (number), `unitPriceCents` (number)
- `userId` (string, optional) тАФ UUID of authenticated user
- `guestEmail` (string, optional) тАФ email for guest checkout
- `orderType` (string, required) тАФ `one_time` or `subscription`

**Validation constraint check order** (perform in this exact sequence to return the most actionable error first):

1. Look up the code (case-insensitive). If not found OR `is_active = false` тЖТ return `NOT_FOUND`: "This promo code is invalid or has expired."
2. Check `expires_at`. If past тЖТ return `EXPIRED`: "This promo code expired on [formatted date]."
3. Check `redemption_count >= max_uses` (when max_uses is not null) тЖТ return `EXHAUSTED`: "This promo code has reached its usage limit."
4. Check per-customer usage by querying `qb_promo_code_redemptions` for the user's ID or email. If `>= max_uses_per_customer` тЖТ return `ALREADY_USED_MAX_TIMES`: "You have already used this code the maximum number of times."
5. Check `restricted_to_email`. If set and doesn't match the authenticated user's email or guest email тЖТ return `EMAIL_RESTRICTED`: "This promo code is not valid for your account." (generic message тАФ do NOT reveal the restricted email)
6. Check `product_ids` and `category_ids`. If set and none of the order items match тЖТ return `PRODUCT_MISMATCH`: "This code is not valid for the selected product(s)."
7. Check `min_order_amount_cents`. If order subtotal is below тЖТ return `MINIMUM_NOT_MET`: "This code requires a minimum order of $[amount] CAD. Your current subtotal is $[amount]."
8. Check `first_time_customer_only`. Query `qb_orders` for any completed order by the user/email. If exists тЖТ return `FIRST_TIME_ONLY`: "This promo code is for first-time customers only."
9. All checks pass тЖТ compute discount amount and return success.

**Success response:**
- `valid`: true
- `discountType`: the code's type
- `discountValue`: the percentage or cent amount
- `discountAmountCents`: the computed discount for this specific order
- `appliesToItems`: which order items receive the discount
- `stackable`: whether this code stacks with the launch promo
- `finalOrderTotalCents`: the final total after discount (and after launch promo if applicable)
- `previewLineItems`: array of line items showing each discount separately
- `codeDescription`: optional display-friendly label (NOT the internal description тАФ generate one like "25% off" or "$25.00 off")

**Failure response:** `{ valid: false, errorCode: string, errorMessage: string }`

**Launch promo stacking logic:**
- Read a `LAUNCH_PROMO_ACTIVE` environment variable (boolean). If active, the automatic 50% launch promo is in effect.
- If the code has `stackable_with_launch_promo = true`: apply launch promo first (50% off base price), then apply code discount to the post-promo price (unless `applies_to_base_price = true`, in which case apply code to base price).
- If `stackable_with_launch_promo = false`: compute both discounts, apply whichever is greater. Include a notice in the response explaining which was applied and why.

**Rate limiting:** This endpoint must be rate-limited to 10 requests per minute per IP address to prevent code enumeration attacks. Use the existing `express-rate-limit` setup pattern from the project.

> **тЪая╕П DO NOT:**
> - Increment `redemption_count` in the validate endpoint тАФ validation is read-only
> - Return the code's `description`, `restricted_to_email`, or `owner_user_id` in the response
> - Use different error messages for non-existent vs deactivated codes
> - Accept codes with fewer than 6 characters тАФ reject at the input validation level
> - Calculate monetary values with floating point тАФ use integer cents throughout

### 2b. `POST /api/qb/promo/redeem`

This endpoint is called server-side at order confirmation time (after Stripe payment succeeds or for $0 order bypass). It atomically redeems the code.

**This endpoint must:**
1. Re-validate ALL constraints (same checks as validate тАФ a race condition may have changed state between validate and redeem)
2. Use a database transaction with `SELECT FOR UPDATE` on the promo code row to prevent double-redemption
3. Atomically increment `redemption_count` on the `qb_promo_codes` record
4. Create a `qb_promo_code_redemptions` record with all required fields
5. If the code has an `owner_user_id` (referral code): create a `qb_referral_credits` record for the referring subscriber
6. Apply the Stripe Coupon to the PaymentIntent or Subscription (for non-$0 orders)
7. Return the final confirmed discount amount

**Race condition handling:** If the usage limit is reached between validate and redeem (concurrent redemptions), return a graceful error: "This promo code was just used by another customer and has reached its limit. Please complete your order at the standard price." The order must NOT be cancelled тАФ the customer proceeds without the discount.

**For $0 orders:** Do NOT create a Stripe PaymentIntent. Record the discount in the redemptions table only. The order's `payment_status` must be set to `free_promo`.

> **тЪая╕П DO NOT:**
> - Skip the re-validation step тАФ all constraints must be checked again inside the transaction
> - Forget the `SELECT FOR UPDATE` тАФ without it, two concurrent requests can both succeed on a single-use code
> - Call Stripe APIs for $0 orders тАФ there is no payment to process

## Step 3: Stripe Coupon Integration

### 3a. Create Stripe Coupon and Promotion Code

When a promo code is created (admin creation will be in Prompt 10, but the utility function is needed now for referral auto-generation), create both a Stripe `Coupon` and a Stripe `PromotionCode` object.

Create a utility module `artifacts/api-server/src/lib/promo-stripe.ts` that exports helper functions:

- `createStripeCouponAndPromoCode(codeData)` тАФ creates the Stripe Coupon (with `percent_off` or `amount_off` depending on type, `currency: 'cad'` for fixed amounts, `duration` and `duration_in_months` for subscription types, `max_redemptions` matching `max_uses`, `redeem_by` matching `expires_at`), then creates a Stripe PromotionCode linked to the Coupon with the code string. Returns both Stripe IDs.

- `deactivateStripePromoCode(stripePromotionCodeId)` тАФ calls `stripe.promotionCodes.update(id, { active: false })`.

- `applyStripeCouponToPayment(stripeCouponId, paymentIntentId)` тАФ applies the coupon to the PaymentIntent via the `discounts` parameter.

- `applyStripeCouponToSubscription(stripeCouponId, subscriptionId)` тАФ applies the coupon to the Subscription.

**All Stripe calls must use idempotency keys** derived from `promoCodeId + orderId` to prevent duplicate discount application on retry.

**If Stripe keys are not configured** (dev mode without Stripe), log a warning and skip the Stripe operations. The database-side promo code logic must still work without Stripe.

> **тЪая╕П DO NOT:**
> - Use `amount_off` without specifying `currency: 'cad'` тАФ Stripe requires it for fixed-amount coupons
> - Create a Coupon without setting `max_redemptions` when `max_uses` is set тАФ Stripe and the database must be in sync
> - Forget idempotency keys on Stripe calls тАФ retries must not create duplicate coupons

## Step 4: Referral Code Auto-Generation

### 4a. Referral Code Generator Utility

Create a utility function that generates unique referral codes. The code format is `REF` followed by 6 uppercase alphanumeric characters, using a readable charset that excludes easily confused characters (O, 0, I, 1). The function must attempt up to 5 times to generate a unique code, checking the database for collisions. If all 5 attempts collide, throw an error.

### 4b. Wire Into Premium Subscription Activation

In the existing subscription webhook handler (in `qb-subscriptions.ts`), when a `checkout.session.completed` event activates a Premium subscription:

1. Check if the subscriber already has a referral code in `qb_promo_codes` with their `owner_user_id`. If yes, reactivate it (set `is_active = true`) instead of creating a new one.
2. If no existing code, generate a new one using the utility from 4a.
3. Insert into `qb_promo_codes` with: `type = fixed_amount`, `amount_off_cents = 2500` ($25 CAD), `max_uses = null` (unlimited total), `max_uses_per_customer = 1`, `stackable_with_launch_promo = true`, `category_ids = null` (all products), `description = 'Referral code тАФ [subscriber name or email]'`, `owner_user_id` = subscriber's user ID.
4. Also create the corresponding Stripe Coupon and PromotionCode via the utility from Step 3.
5. Store the code on the subscriber's record so it can be displayed in their portal Settings > My Plan section.

### 4c. Referral Credit on Redemption

In the redemption endpoint (Step 2b), when a referral code is redeemed (identified by `owner_user_id` being set on the code):

1. Create a `qb_referral_credits` record: `beneficiary_user_id` = the code's `owner_user_id`, `amount_cents = 2500`, `source = 'promo_code_redemption'`, `promo_code_redemption_id` = the redemption record ID, `order_id` = the order, `status = 'pending'`.
2. This credit is tracked and displayed to the subscriber but NOT automatically applied to invoices at this stage (manual application by Hassan is the v1 approach).

> **тЪая╕П DO NOT:**
> - Generate a new referral code if one already exists for the subscriber тАФ reactivate the existing one
> - Make referral codes single-use globally тАФ they are `max_uses = null` (unlimited) but `max_uses_per_customer = 1`
> - Auto-apply referral credits to invoices тАФ that is a v2 feature
