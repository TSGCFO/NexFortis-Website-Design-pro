`But Render static sites don't support proxying, so you'd need all three as web services or use a separate reverse proxy.`

That statement is FALSE. Here's the proof straight from Render's official documentation.
The evidence
Render's docs page "Static Site Redirects and Rewrites" ([https://render.com/docs/redirects-rewrites](https://render.com/docs/redirects-rewrites)) explicitly states:
"Destination can be either a path or a full, publicly accessible URL."
And it gives this exact example in their official table of basic examples:
SourceDestination
/web-host
[https://render.com](https://render.com)
That's a rewrite rule on a static site whose destination is an external full URL — exactly the pattern I described for proxying /api/* to your backend.
It also explains the difference between Redirect and Rewrite, confirming that Rewrite "does not redirect the browser. Instead, your site serves the content from the rule's destination at the original path. The browser can't detect that content was served from a different path or URL." That is the textbook definition of a reverse proxy, and it's a built-in feature of Render static sites.
So in your specific case
A rewrite rule on the qb-portal static site like:
SourceDestinationAction
/api/*
https://nexfortis-api.onrender.com/api/*
Rewrite
is fully supported, officially documented, and works exactly as I described. The visitor's browser sees the response come from qb.nexfortis.com/api/... — same origin — while Render quietly fetches it from your backend service behind the scenes. No CORS issue, no separate reverse proxy needed, no need to make the frontends web services.
One small caveat worth noting
The docs do say: "You can't apply redirect/rewrite rules to your domain root." That just means Source must include at least one path component (like /api/* or /*). It does not restrict the destination from being an external URL. Your /api/* rule satisfies that requirement just fine.
So you can confidently proceed with the three-service architecture I laid out: one paid web service for the API, two free static sites for the frontends, with /api/* rewrites pointing the frontends at the API. That setup is exactly what Render's product was designed to support.
