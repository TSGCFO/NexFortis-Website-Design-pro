## Fix 3 вЂ” IMPORTANT: Subscriber discount should stack with promo codes (IMP-01)

In `artifacts/api-server/src/routes/qb-portal.ts`, the checkout endpoint (`/checkout/create-session`) currently applies subscriber discount ONLY when no promo code is present. Per the PRD, subscriber discounts are automatic tier-based discounts that always apply, and promo codes stack on top.

Find the block that starts with:

```
// Subscriber tier discount: only for non-subscription orders, and only when no promo code applied
let subscriberDiscountCents = 0;
if (!promoRow && userId && !pricing.isSubscription) {
```

Change the condition from `if (!promoRow && userId && !pricing.isSubscription)` to:

```
if (userId && !pricing.isSubscription) {
```

Remove the `!promoRow &&` condition so subscriber discounts are always computed.

Then, the promo code validation block (which runs before the subscriber discount block) needs to be aware of the subscriber discount. Reorder the logic so that:

1. First, compute the subscriber discount on `pricing.total`
2. Then, compute the promo code discount on the post-subscriber-discount subtotal

To do this:
1. Move the subscriber discount block ABOVE the promo code validation block
2. In the promo validation block, pass the subscriber-adjusted total as `unitPriceCents` in the `orderItems` array, not the original `pricing.total`
3. Update `serverComputedFinalTotal` to account for both discounts

The final `serverComputedFinalTotal` should be:
```
serverComputedFinalTotal = pricing.total - subscriberDiscountCents - serverDiscountCents
```

Ensure the `totalDiscount` variable (used to set `discountAmountAppliedCents`) sums both:
```
const totalDiscount = serverDiscountCents + subscriberDiscountCents;
```

This is already the case вЂ” just verify it still works after reordering.

> **вљ пёЏ DO NOT:**
> - Apply subscriber discount for subscription orders вЂ” the `!pricing.isSubscription` guard must stay
> - Apply subscriber discount to the base price before launch promo вЂ” the `runValidation` function already handles launch promo stacking internally

## Fix 4 вЂ” IMPORTANT: runValidation called outside transaction (IMP-02)

In `artifacts/api-server/src/routes/qb-promo.ts`, inside the `runRedemption` function, the `runValidation()` call at line ~396 executes outside the transaction context. The critical race-condition checks (max uses, per-customer limit) should use the transaction.

Replace the call to `runValidation(...)` inside the transaction with inline checks using the `tx` parameter for the per-customer and first-time-customer validations. Specifically:

1. Keep the `SELECT FOR UPDATE` lock and the initial checks on the locked row (isActive, expiresAt, maxUses) вЂ” these are already correct.

2. Replace the `runValidation(...)` call with direct checks using `tx` instead of `db`:
   - Per-customer usage check: query `qb_promo_code_redemptions` using `tx` for the user/email count
   - Email restriction check: query `qb_users` using `tx` to get the authenticated user's email
   - Product match check: call `productMatches(row, items)` directly (this is a pure function, no DB)
   - Min order amount check: call `computeBaseSubtotal(items)` directly (pure function)
   - First-time customer check: query `qb_orders` using `tx` for previous completed orders

3. After all checks pass, compute the discount using the existing pure functions (`computeDiscount`, `launchPromoActive`, etc.)

This ensures all validation reads happen within the same transaction, under the same lock.

> **вљ пёЏ DO NOT:**
> - Remove the `SELECT FOR UPDATE` вЂ” it is critical for preventing double-redemption
> - Remove the re-validation entirely вЂ” the locked row values must be re-checked
> - Change the external `runValidation` function вЂ” other callers (validate endpoint) use it correctly
