## Step 3: Subscription API Routes

Create subscription-related routes. These go in a new file `artifacts/api-server/src/routes/qb-subscriptions.ts` to keep qb-portal.ts from getting too large. Register the new router in the main router/app.

All subscription routes are prefixed with `/api/qb/subscriptions/`.

### 3a. Create Subscription Checkout Session

`POST /api/qb/subscriptions/checkout` — requires `requireAuth`

Request body: `{ tier: "essentials" | "professional" | "premium" }`

Logic:
1. Validate tier is one of the three allowed values. Return 400 for invalid.
2. Check if user already has an active subscription. Return 409 with `{ error: "You already have an active subscription", currentTier: "..." }`.
3. Look up or create a Stripe customer for this user (by email). Store `stripe_customer_id` on `qb_users` if not already set.
4. Determine which price to use: if promo is active (check a config flag or env var `PROMO_ACTIVE=true`), use the promo price. Otherwise, use standard price.
5. Create a Stripe Checkout Session in `subscription` mode:
   - `customer`: the Stripe customer ID
   - `line_items`: the selected price
   - `mode`: `"subscription"`
   - `metadata`: `{ user_id: req.userId, tier }`
   - `success_url`: `{origin}/qb-portal/subscription?success=true`
   - `cancel_url`: `{origin}/qb-portal/subscription?canceled=true`
   - `subscription_data.metadata`: `{ user_id: req.userId, tier }`
   - `allow_promotion_codes`: false (we manage promos ourselves)
   - Tax: if Stripe Tax is configured, enable `automatic_tax: { enabled: true }`. Otherwise, skip for now — tax integration is a later concern.
6. Return `{ checkoutUrl: session.url }`

### 3b. Get Current Subscription

`GET /api/qb/subscriptions/me` — requires `requireAuth`

Returns the user's active subscription (if any) with:
- `subscription` object (tier, status, currentPeriodStart, currentPeriodEnd, cancelAtPeriodEnd)
- `ticketUsage` for the current billing cycle (ticketsUsed, ticketLimit, healthCheckUsed, videoCallUsed)
- `referral` info if Premium (referralCode, totalEarnings)

If no subscription exists, return `{ subscription: null }`.

**Ticket limits by tier:**
- Essentials: 3
- Professional: 8
- Premium: Infinity (return `null` for limit)

### 3c. Upgrade Subscription

`POST /api/qb/subscriptions/upgrade` — requires `requireAuth`

Request body: `{ newTier: "professional" | "premium" }`

Logic:
1. Get user's current subscription. Return 404 if none.
2. Validate newTier is higher than current tier (essentials < professional < premium). Return 400 if not an upgrade.
3. Use Stripe API to update the subscription's price:
   - `stripe.subscriptions.update(stripeSubscriptionId, { items: [{ id: existingItemId, price: newPriceId }], proration_behavior: 'create_prorations' })`
4. Update the local DB record (tier, stripePriceId, updatedAt).
5. Return the updated subscription.

### 3d. Downgrade Subscription

`POST /api/qb/subscriptions/downgrade` — requires `requireAuth`

Request body: `{ newTier: "essentials" | "professional" }`

Logic:
1. Get user's current subscription. Return 404 if none.
2. Validate newTier is lower than current tier. Return 400 if not a downgrade.
3. Schedule the downgrade for period end using Stripe's subscription schedule or `cancel_at_period_end` + re-create pattern. Simplest approach:
   - Use `stripe.subscriptions.update(stripeSubscriptionId, { items: [{ id: existingItemId, price: newPriceId }], proration_behavior: 'none', billing_cycle_anchor: 'unchanged' })` — this changes the price at next renewal.
   - OR schedule via Stripe Subscription Schedules API.
4. Update local DB: set a `pendingTier` or note the scheduled change. The webhook for `customer.subscription.updated` will sync the actual tier when Stripe processes it.
5. Return confirmation with effective date (current period end).

### 3e. Cancel Subscription

`POST /api/qb/subscriptions/cancel` — requires `requireAuth`

Logic:
1. Get user's current subscription. Return 404 if none.
2. Call `stripe.subscriptions.update(stripeSubscriptionId, { cancel_at_period_end: true })`.
3. Update local DB: set `cancelAtPeriodEnd = true`, `status = 'cancel_at_period_end'`.
4. Return `{ message: "Subscription will remain active until {periodEnd}. No further charges will be made." }`.

### 3f. Reactivate Subscription (undo cancel)

`POST /api/qb/subscriptions/reactivate` — requires `requireAuth`

Logic:
1. Get user's subscription where `cancelAtPeriodEnd = true`. Return 404 if not in cancel state.
2. Call `stripe.subscriptions.update(stripeSubscriptionId, { cancel_at_period_end: false })`.
3. Update local DB: set `cancelAtPeriodEnd = false`, `status = 'active'`.
4. Return updated subscription.

**Verify Step 3:** `pnpm typecheck` — zero errors.

## Step 4: Expand Stripe Webhook Handler

Update the existing webhook handler in `qb-portal.ts` (or create a shared webhook file) to handle subscription events. The webhook must handle these events IN ADDITION to the existing `checkout.session.completed` for one-time orders:

### 4a. `checkout.session.completed` (subscription mode)

When `session.mode === "subscription"`:
1. Extract `userId` and `tier` from `session.metadata`.
2. Get the Stripe subscription ID from `session.subscription`.
3. Retrieve the full subscription from Stripe: `stripe.subscriptions.retrieve(subscriptionId)`.
4. Create a `qb_subscriptions` record with all fields.
5. Create the first `qb_ticket_usage` record for the current billing cycle.
6. If tier is "premium", generate a referral code (8 chars, alphanumeric, cryptographically random) and create a `qb_referrals` record.
7. Update `qb_users.stripe_customer_id` if not already set.

**Idempotency:** Check if a subscription record with this `stripe_subscription_id` already exists before inserting.

### 4b. `invoice.paid`

1. Find the subscription by `invoice.subscription`.
2. Update `current_period_start` and `current_period_end` from the invoice's period.
3. Set status to `'active'` (clears any `past_due`).
4. Create a new `qb_ticket_usage` record for the new billing cycle (reset ticket count to 0, health_check_used to false, video_call_used to false). Only create if one doesn't exist for this cycle_start already.

### 4c. `invoice.payment_failed`

1. Find the subscription by `invoice.subscription`.
2. Set status to `'past_due'`.

### 4d. `customer.subscription.updated`

1. Find the subscription by the Stripe subscription ID.
2. Sync: `tier` (derived from the current price ID → tier mapping), `status`, `cancelAtPeriodEnd`, `currentPeriodStart`, `currentPeriodEnd`, `stripePriceId`.
3. For tier mapping: maintain a map of Stripe Price IDs to tier names. Read from env vars or a config object.

### 4e. `customer.subscription.deleted`

1. Find the subscription by the Stripe subscription ID.
2. Set `status = 'canceled'`.

> **⚠️ COMMON MISTAKE — DO NOT:**
> - Process the same webhook event twice — check by event ID or subscription state
> - Assume `checkout.session.completed` is always for one-time orders — check `session.mode`
> - Crash on unknown event types — log and return 200 for any unhandled event type

## Step 5: Ticket Submission with Subscription Awareness

Update the existing ticket submission route (or create a new one if it doesn't exist) to be subscription-aware.

### 5a. Ticket Submission Route

`POST /api/qb/tickets` — requires `requireAuth`

Request body: `{ subject, message, isCritical?: boolean, attachmentFile?: multipart }`

Logic:
1. Check if user has an active subscription.
2. If subscriber:
   a. Look up the current `qb_ticket_usage` record.
   b. Check ticket limit: Essentials=3, Professional=8, Premium=unlimited.
   c. If at limit, return 403 with `{ error: "ticket_limit_reached", ticketsUsed: N, ticketLimit: N, resetDate: "...", upgradeTier: "..." }`.
   d. Increment `tickets_used` on the usage record.
   e. Set `tier_at_submission` and `subscription_id` on the ticket.
3. If NOT a subscriber: still allow ticket submission (non-subscribers can submit general tickets, but without SLA guarantees). Set `tier_at_submission = null`.
4. Determine if `is_critical`: either `isCritical` is true from the request, OR the message body contains keywords: "error -6000", "won't open", "data loss", "corrupt", "can't open company file".
5. Determine if `is_after_hours`: check if current time (America/Toronto timezone) is outside Mon–Fri 9am–5pm ET.
6. Calculate `sla_deadline` using the business-hours logic from Step 6.
7. If attachment file is provided, upload to Supabase Storage under `tickets/{userId}/{ticketId}/` and store the path.
8. Insert the ticket record.
9. Return the created ticket with `{ ticketsRemaining }`.

### 5b. Get User's Tickets

`GET /api/qb/tickets` — requires `requireAuth`

Returns the authenticated user's tickets, ordered by created_at desc. Include: id, subject, status, isCritical, createdAt, firstResponseAt, tierAtSubmission.

### 5c. Get Ticket Detail

`GET /api/qb/tickets/:id` — requires `requireAuth`

Returns full ticket including message body, operator reply, internal note (only if user is operator), attachment URL (Supabase signed URL if attachment exists).

Verify the ticket belongs to the requesting user (or user is operator).

## Step 6: SLA Timer and Business Hours Logic

Create a utility module at `artifacts/api-server/src/lib/business-hours.ts`.

### 6a. Business Hours Definition

- Business hours: Monday–Friday, 9:00 AM – 5:00 PM Eastern Time (America/Toronto)
- Holidays: not tracked in v1 (all weekdays are business days)

### 6b. Functions to Implement

**`isBusinessHours(date: Date): boolean`**
Returns true if the given date falls within business hours.

**`calculateSlaDeadline(submissionTime: Date, slaMinutes: number, isCritical: boolean): Date`**
- If `isCritical`: deadline = submissionTime + slaMinutes (calendar minutes, NOT business hours). Critical tickets always have a 60-minute window regardless of time of day.
- If NOT critical: deadline = submissionTime + slaMinutes, counting only business-hour minutes. If submitted at 4:30 PM ET with a 60-minute SLA, the deadline is 9:30 AM ET the next business day (30 minutes remaining carry to next day).

**`getSlaMinutesForTier(tier: string | null): number`**
- `essentials`: 60
- `professional`: 60
- `premium`: 30
- `null` (non-subscriber): no SLA (return 0 or null)

**`getTimeRemainingMinutes(slaDeadline: Date): number`**
Returns minutes remaining, counting only business-hour minutes. Used by the admin dashboard to display the countdown timer.

**`getSlaStatus(slaDeadline: Date): 'green' | 'yellow' | 'red' | 'breached'`**
- `green`: > 30 minutes remaining
- `yellow`: 10–30 minutes remaining
- `red`: < 10 minutes remaining
- `breached`: deadline has passed

### 6c. Edge Cases

- Ticket submitted Friday at 4:50 PM ET with 60-min SLA: 10 minutes count Friday, 50 minutes carry to Monday 9:00–9:50 AM ET. Deadline: Monday 9:50 AM ET.
- Ticket submitted Saturday: SLA starts Monday 9:00 AM ET.
- Ticket submitted at 3:00 AM ET on Wednesday: SLA starts Wednesday 9:00 AM ET.
- Critical ticket submitted Saturday at midnight: deadline is Saturday 1:00 AM (60 calendar minutes, no business-hours adjustment).

**Verify Step 6:** Write a few inline test cases or log outputs to confirm the business-hours calculations are correct. Test at least: mid-business-hours, end-of-day carryover, weekend submission, critical ticket.
