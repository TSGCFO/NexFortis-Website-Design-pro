## Fix 1: isQbmProduct defaults to true when product not found

**File:** `artifacts/api-server/src/routes/qb-portal.ts`, around line 491


## Fix 4: Upload-token not validated before file parsing

**File:** `artifacts/api-server/src/routes/qb-portal.ts`, the `router.post("/orders/:id/files", ...)` handler around line 423

**Current code:**
The route checks for auth headers (Bearer or x-upload-token), but the upload-token is only validated against the database AFTER multer processes the multipart body. This means an invalid upload-token still causes the server to buffer the entire file before returning 401/403.

**Fix:** Add a pre-multer validation step. Before calling `singleUpload(req, res, ...)`, if the request uses `x-upload-token` auth (no Bearer header), do a quick database check to verify the token matches the order. If it doesn't match, return 401 immediately without parsing the file.

The flow should be:
1. Parse orderId from params
2. Check which auth method is being used
3. If Bearer token: proceed to multer (JWT validation happens after)
4. If x-upload-token: query the database to verify `qbOrders` where `id = orderId AND uploadToken = uploadTokenHeader`. If no match, return `401 { error: "Invalid upload token" }` immediately. If match, proceed to multer.
5. If neither: return 401 (this check already exists)

This prevents wasting bandwidth and memory on unauthorized uploads.

## Fix 5: pg_cron DELETE + UPDATE not in a transaction

**File:** `artifacts/api-server/src/migrations/supabase-storage-cron.sql`

**Current code:**
The cron job has two separate SQL statements: a DELETE from `storage.objects` and an UPDATE of `public.qb_order_files`. If one succeeds and the other fails, data is inconsistent (e.g., files deleted from storage but not marked expired in the database).

**Fix:** Wrap both statements in a transaction block. Change the cron body from:
```sql
-- Step 1: Delete storage objects older than 7 days from the order-files bucket
DELETE FROM storage.objects
WHERE bucket_id = 'order-files'
  AND created_at < NOW() - INTERVAL '7 days';

-- Step 2: Mark qb_order_files records as expired (only those not already expired)
UPDATE public.qb_order_files
SET expired = true,
    deleted_at = NOW()
WHERE uploaded_at < NOW() - INTERVAL '7 days'
  AND storage_path IS NOT NULL
  AND expired = false;
```

To:
```sql
BEGIN;

-- Step 1: Delete storage objects older than 7 days from the order-files bucket
DELETE FROM storage.objects
WHERE bucket_id = 'order-files'
  AND created_at < NOW() - INTERVAL '7 days';

-- Step 2: Mark qb_order_files records as expired (only those not already expired)
UPDATE public.qb_order_files
SET expired = true,
    deleted_at = NOW()
WHERE uploaded_at < NOW() - INTERVAL '7 days'
  AND storage_path IS NOT NULL
  AND expired = false;

COMMIT;
```

## Fix 7: Admin layout no cleanup on unmount

**File:** `artifacts/qb-portal/src/components/admin-layout.tsx`, around lines 29-53

**Current code:**
The async IIFE inside useEffect has no cleanup. If the component unmounts while the async call is in flight, React will try to set state on an unmounted component.

**Fix:** Add a cleanup flag to the useEffect. Add `let cancelled = false;` at the top of the effect, check `if (cancelled) return;` before each `setState` call inside the async IIFE, and return a cleanup function `return () => { cancelled = true; };` from the useEffect.

The pattern:
```
useEffect(() => {
  if (authLoading) return;
  // ... early returns for !session, !isOperator ...

  let cancelled = false;
  (async () => {
    setChecking(true);
    try {
      // ... existing logic ...
      if (cancelled) return;
      setAal2Verified(true);
    } catch {
      if (cancelled) return;
      navigate("/admin/mfa-challenge");
    } finally {
      if (!cancelled) setChecking(false);
    }
  })();

  return () => { cancelled = true; };
}, [authLoading, session, user, isOperator, navigate]);
```

## Fix 10: Challenge page doesn't check nextLevel

**File:** `artifacts/qb-portal/src/pages/admin/mfa-challenge.tsx`, around line 41

**Current code:**
```
if (aalData.currentLevel === "aal2") {
  setState("redirecting");
  navigate("/admin");
  return;
}
```

Only checks `currentLevel` but not `nextLevel`. The Supabase AAL response includes both `currentLevel` and `nextLevel`. If `nextLevel === "aal2"` but `currentLevel !== "aal2"`, the user needs to complete MFA. But if somehow `currentLevel === "aal2"` and `nextLevel` is something unexpected, the redirect might be premature.

**Fix:** Add a check for `nextLevel`. Only redirect to admin if the user truly doesn't need further verification:
```
if (aalData.currentLevel === "aal2" && aalData.nextLevel === "aal2") {
  setState("redirecting");
  navigate("/admin");
  return;
}
```

If `currentLevel` is "aal2" but `nextLevel` indicates more is needed, fall through to the challenge form.
