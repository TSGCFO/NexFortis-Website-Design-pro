## Executive Summary

The NexFortis marketing website is a well-structured React SPA with thoughtful on-page SEO in code — but it has **zero effective search visibility today** due to two compounding production failures:

1. **`nexfortis.com` resolves to a GoDaddy "Launching Soon" placeholder**, not the React app. Google is indexing a bare-bones GoDaddy page titled `nexfortis.com` with the description "Launching Soon". Every proprietary meta tag, JSON-LD schema, and canonical URL in the codebase is irrelevant until DNS is corrected.

2. **`nexfortis-marketing.onrender.com` returns HTTP 404 for every path**, including the root (`/`). The Render static site appears not to have completed a successful deployment. Even if DNS were fixed today, visitors would hit a 404.

Until these two issues are resolved, no other SEO improvement delivers value. The site effectively does not exist on the web.

Beyond the deployment crisis, secondary issues — CSR-only rendering, canonical URL contamination, duplicate JSON-LD schemas, a broken `og:image`, and absent AI-crawler governance — are documented in full below as the next action queue once the site is live.

## Critical Issues

### C-1 — DNS points to GoDaddy placeholder, not Render
**Page:** `https://nexfortis.com/` (all routes)  
**Evidence (curl):**
```
HTTP/1.1 200 OK
Server: DPS/2.0.0+sha-3ef1503
<title>nexfortis.com</title>
<meta name="twitter:description" content="Launching Soon"/>
<meta name="generator" content="Starfield Technologies; Go Daddy Website Builder 8.0.0000"/>
```
All routes except `/` return HTTP 404 from GoDaddy.  
**Impact:** Google has indexed (or will index) a placeholder page with no business content. Every click from Google would bounce on a "Launching Soon" message. All 11 sitemap entries in the Render-side `sitemap.xml` are indexed against the GoDaddy page, which has its own competing `sitemap.xml` pointing only to the homepage.  
**Fix:** Update the `nexfortis.com` A / CNAME records at the registrar to point to Render's static site IP/hostname per the runbook. Remove or transfer the GoDaddy website builder configuration.

**Fix:** Connect the repo to Render via the Blueprint, trigger a manual deploy, and verify the build succeeds. Confirm `artifacts/nexfortis/dist/public/index.html` exists post-build.

## High-Impact Improvements

### H-1 — 100% CSR: per-page meta tags invisible to crawlers on first pass
**Applies to:** All inner pages (`/about`, `/services/*`, `/contact`, `/blog`, `/privacy`, `/terms`)  
**Evidence:** `artifacts/nexfortis/src/main.tsx` mounts `<App />` into `<div id="root"></div>`. There is no SSR. `react-helmet-async` sets all page-specific `<title>`, `<meta>`, OG, canonical, and JSON-LD tags via JavaScript after hydration.  

The static `index.html` contains:
```html
<title>NexFortis IT Solutions — Complexity Decoded. Advantage.</title>
<meta name="description" content="NexFortis delivers end-to-end IT solutions..."/>
<link rel="canonical" href="https://nexfortis.com/" />
```
Googlebot executes JavaScript (second-wave indexing, typically delayed hours to days), so page-specific meta will eventually be seen, but:
- First-wave HTML crawls of all inner pages return the homepage shell meta
- All inner pages share the same raw-HTML title/description temporarily
- Rendered-meta canonicals may point to the wrong domain (see H-2)
- This creates duplicate-content signals across all inner pages during the indexing delay window

**Fix (longer term):** Implement SSR or static pre-rendering (e.g., Vite SSG with `vite-plugin-ssr` or move to Remix/Next.js). **Fix (short term / quick-win bridge):** Pre-bake critical per-page `<title>` and `<meta name="description">` overrides directly into `index.html` using Vite's `html` plugin transform or a build-time HTML rewriter for the most important routes.

### H-2 — `getSiteUrl()` injects Render subdomain into canonical tags
**Applies to:** All inner pages when served from `nexfortis-marketing.onrender.com`  
**Evidence (`seo.tsx` lines 16–21):**
```ts
function getSiteUrl(): string {
  if (typeof window !== "undefined" && window.location.hostname !== "localhost") {
    return window.location.origin;  // ← returns the Render subdomain at runtime
  }
  return SITE_URL;  // "https://nexfortis.com"
}
```
When a user (or Googlebot) reaches the site via `nexfortis-marketing.onrender.com`, every canonical tag becomes:
```html
<link rel="canonical" href="https://nexfortis-marketing.onrender.com/about" />
```
instead of:
```html
<link rel="canonical" href="https://nexfortis.com/about" />
```
Google would treat the Render subdomain as the canonical — potentially ranking the Render URL instead of `nexfortis.com`, or splitting link equity between the two domains.  
**Fix:** Replace the dynamic `getSiteUrl()` logic with a build-time constant. Either hardcode `SITE_URL = "https://nexfortis.com"` and remove the `window.location.origin` branch, or inject it via the `VITE_SITE_URL` environment variable at build time so it is correct in every deployment context.

### H-3 — Duplicate JSON-LD schemas on the homepage
**Applies to:** `/` (homepage)  
**Evidence:** The static `index.html` includes two `<script type="application/ld+json">` blocks (a `LocalBusiness` and a `WebSite` schema), both injected at build time. The homepage React component (`home.tsx`) additionally renders `<OrganizationSchema />`, `<LocalBusinessSchema />`, and `<WebSiteSchema />` via `react-helmet-async`, which injects three more schema blocks into `<head>` at runtime.

Result: the homepage has **6 JSON-LD blocks total** — two from the HTML shell (`LocalBusiness` + `WebSite`) and four from React (`<OrganizationSchema />`, `<LocalBusinessSchema />`, `<WebSiteSchema />`, and `<FAQSchema faqs={homeFaqs} />`). Both `LocalBusiness` and `Organization` schemas share `@id: siteUrl/#organization`, creating an `@id` collision that is invalid in JSON-LD. Google's rich results validator may reject the structured data or produce unpredictable results.

**Fix:** Given that the site is 100% CSR (H-1), the static JSON-LD in `index.html` is the only schema visible to crawlers on the first HTML pass — it should be **kept and corrected**, not removed. The correct fix is to remove the three React schema component calls (`<OrganizationSchema />`, `<LocalBusinessSchema />`, `<WebSiteSchema />`) from `home.tsx` and let the static `index.html` blocks serve as the sole homepage schemas. This eliminates the duplication and the `@id` collision, while preserving crawler-visible structured data. Any dynamic schema components that are kept should use the correct, non-colliding `@id` values (e.g., `@id: siteUrl/#website` for the WebSite schema). Note: once H-2 is fixed (canonical URLs hardcoded), the static `index.html` schemas will also have correct, non-environment-dependent URLs.

### H-4 — `WebSite` schema `SearchAction` target is invalid
**Evidence (`seo.tsx` line 150):**
```json
"potentialAction": {
  "@type": "SearchAction",
  "target": "https://nexfortis.com/blog?q={search_term_string}",
  "query-input": "required name=search_term_string"
}
```
The blog page at `/blog` does not implement URL-based search (`?q=`). There is no server-side or client-side handler for this query parameter. Google will attempt to validate the SearchAction endpoint and find no response, causing the Sitelinks Searchbox to be dropped or never triggered.  
**Fix:** Either implement genuine blog search (server-side `GET /api/blog/posts?q=…` with client-side URL sync) or remove the `potentialAction` from the `WebSite` schema entirely.

### H-5 — Admin login page is not blocked in robots.txt; noindex relies solely on JS execution
**Applies to:** `/admin/login`  
**Evidence (`robots.txt`):**
```
User-agent: *
Allow: /
Disallow: /blog/admin
Sitemap: https://nexfortis.com/sitemap.xml
```
**Evidence (`pages/admin-login.tsx` line 39):**
```tsx
<SEO title="Operator Login" description="Operator login for NexFortis blog admin." path="/admin/login" noIndex />
```
`/admin/login` is not blocked by `robots.txt` and is not included in `sitemap.xml`. The component does render `<SEO noIndex />` which sets `<meta name="robots" content="noindex,nofollow" />` — but only after JavaScript executes. Since the site is 100% CSR (H-1), a crawler that fetches the raw HTML shell (`index.html`) before running JS will receive the default `index, follow` signal from the static `<meta name="robots" content="index, follow" />` in `index.html`. The noindex is therefore contingent on the crawler rendering JS, which is not guaranteed on every crawl pass.  
**Fix:** Add `Disallow: /admin/login` to `robots.txt` as a robust, JS-independent block. The existing `<SEO noIndex />` in the component remains as correct defence-in-depth once the site is rendered.

### H-6 — No AI crawler governance in robots.txt
**Evidence (`public/robots.txt`):**
```
User-agent: *
Allow: /
Disallow: /blog/admin
Sitemap: https://nexfortis.com/sitemap.xml
```
No explicit rules for LLM training crawlers (GPTBot, Claude-Web / anthropic-ai, CCBot, PerplexityBot) or search-AI crawlers (Applebot, Bingbot for Copilot grounding). As AI-powered SERPs become a dominant traffic source, controlling which crawlers may use the content for training vs. grounding is increasingly important.  
**Impact:** Content may be used for model training without explicit consent. Conversely, blocking grounding crawlers reduces visibility in AI-generated answers.  
**Fix:** Add explicit agent stanzas per business policy. Suggested baseline:
```
# AI training crawlers — disallow training use, allow search/grounding
User-agent: GPTBot
Disallow: /

User-agent: anthropic-ai
Disallow: /

User-agent: CCBot
Disallow: /

# AI search grounding — allow (improves AI-answer visibility)
User-agent: PerplexityBot
Allow: /

User-agent: Applebot
Allow: /
```

## Quick Wins

### Q-1 — Missing `og:image:width` / `og:image:height` meta tags
**Applies to:** All pages  
**Evidence:** Neither `index.html` nor `seo.tsx` specify OG image dimensions.  
**Fix:** Add to `index.html` (and to the `SEO` component's Helmet output):
```html
<meta property="og:image:width" content="1200" />
<meta property="og:image:height" content="630" />
<meta property="og:image:type" content="image/png" />
```
Verify the actual dimensions of `opengraph.png` (should be 1200×630 for best results on LinkedIn, Slack, Twitter/X). Note: `opengraph.jpg` also exists in the public directory — consolidate to one file and update references.

### Q-2 — Missing `twitter:site` handle
**Evidence (`seo.tsx` line 47–50):** Twitter Card tags do not include `twitter:site`.  
**Fix:** Add `<meta name="twitter:site" content="@NexFortisIT" />` (or the correct handle) to the SEO component and `index.html`.

### Q-3 — Sitemap lacks `<lastmod>` dates
**Evidence (`public/sitemap.xml`):** No `<lastmod>` on any URL entry.  
**Fix:** Add `<lastmod>` dates. For static pages, use the last known significant content change date. For blog, generate the sitemap dynamically via the API.

---

### Q-4 — Blog post URLs missing from sitemap
**Evidence (`sitemap.xml`):** Only `/blog` is listed, not individual post URLs.  
**Impact:** Individual blog posts (which are the highest-value long-tail SEO targets) are not directly submitted to Google.  
**Fix:** Generate a dynamic sitemap at `/sitemap.xml` via the API server (`GET /api/sitemap`) that includes static routes plus all published blog post slugs with their `createdAt` as `<lastmod>`.

---

### Q-5 — `manifest.json` icon purpose is malformed
**Evidence (`public/manifest.json`):**
```json
{
  "src": "icon-512.png",
  "sizes": "512x512",
  "type": "image/png",
  "purpose": "any maskable"
}
```
W3C Web App Manifest spec requires `purpose` to contain one value per entry. `"any maskable"` is technically valid (space-separated list is allowed per the spec), but Chrome and PWA validators often prefer two separate entries for clarity and compatibility.  
**Fix:**
```json
{ "src": "icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "any" },
{ "src": "icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
```

---

### Q-6 — `ArticleSchema` uses `Article` instead of `BlogPosting`
**Evidence (`seo.tsx` line 239):** `"@type": "Article"` is used for blog posts.  
**Fix:** Change to `"@type": "BlogPosting"` which is the correct schema.org subtype and may unlock richer Google article features.

---

### Q-7 — `ArticleSchema` missing `image` property
**Evidence (`seo.tsx` lines 231–263):** The `ArticleSchema` does not include an `image` field.  
**Impact:** Google's Article rich results require `image` for display in Top Stories / news carousels.  
**Fix:** Pass `coverImage` from the blog post data into `ArticleSchema` and add:
```json
"image": { "@type": "ImageObject", "url": "https://nexfortis.com/images/blog-1.png" }
```

---

### Q-8 — `modulePreload: false` disables Vite's link preload hints
**Evidence (`vite.config.ts` line 62):** `modulePreload: false`  
**Impact:** Vite normally injects `<link rel="modulepreload">` for JS entry points, which gives browsers an early signal to fetch critical scripts in parallel, improving First Contentful Paint and LCP. Disabling this adds a small but measurable latency to JS-heavy routes.  
**Note:** This may have been disabled intentionally (e.g., to avoid a Safari bug or reduce HTML size). If intentional, leave it. Otherwise, remove `modulePreload: false`.

## Technical Foundations

### HTTPS
The production domain (`nexfortis.com`) returns HTTPS with `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload` — **set by GoDaddy**, not by the React app. Render's static sites also serve over HTTPS. HSTS posture is fine; the custom domain needs to be configured in Render's custom domain panel to preserve HTTPS coverage.

### Mobile Viewport
`<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1" />` — present in `index.html`. Note: `maximum-scale=1` prevents zoom on iOS, which can be an accessibility issue and may flag in Google's mobile-friendliness signals. Consider removing `maximum-scale=1`.

### Core Web Vitals Risk Factors (source-level signals)
- **LCP:** Hero image preloaded (`<link rel="preload" href="/images/hero-bg.webp" as="image" />`). Fonts preloaded. Good.
- **LCP risk:** `modulePreload: false` (Q-8) means JS entry chunks are not link-preloaded; this delays Time to Interactive.
- **LCP risk:** All pages load via `React.lazy()` — the initial bundle is lightweight but every page transition triggers a dynamic import waterfall. On slow connections, this can delay LCP on inner pages.
- **CLS risk:** Framer Motion animations use `opacity:0` initial state — ensure elements are not layout-affecting during the animation frame (check `whileInView` usage on service cards).
- **INP:** No obvious blocking scripts. GA4 is deferred behind cookie consent (good for performance).
- **Bundle split:** Vite `manualChunks` correctly separates `framer-motion`, `@tanstack/react-query`, `react-helmet-async`, `lucide-react`, and React vendor — good chunking strategy.
- **Fonts:** `inter-400.woff2` and `alegreya-sans-sc-700-latin.woff2` are preloaded with `crossorigin`. Both use `font-display: swap` (verify in the `@font-face` declarations in the CSS).

### Caching Headers
`render.yaml` sets:
```yaml
- path: /assets/*
  name: Cache-Control
  value: public, max-age=31536000, immutable
```
Good — hashed assets get 1-year immutable cache. No explicit `Cache-Control` on HTML responses (Render default is typically `no-cache` for HTML). This is correct behaviour.

### Security Headers
`render.yaml` sets `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin` for all routes. No `Content-Security-Policy` on the static site (CSP is on the API server only). For a static marketing site, the absence of CSP is a minor security gap but does not directly affect SEO.
